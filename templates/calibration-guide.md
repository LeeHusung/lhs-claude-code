# Evaluator 캘리브레이션 가이드

Evaluator Agent의 채점 정확도를 지속적으로 개선하는 프로세스.
원문 참조: "The fix was an iterative tuning loop."

---

## 왜 캘리브레이션이 필요한가

Claude는 기본적으로 **열악한 QA 에이전트**이다:
- 합법적인 문제를 식별해도 "중요하지 않다"고 스스로 설득
- 표면적 테스트만 수행하고 엣지 케이스를 탐사하지 않음
- 미묘한 버그를 놓침
- 점수를 관대하게 매김

이를 교정하기 위해 **반복적 튜닝 루프**를 실행한다.

---

## 캘리브레이션 루프 (프로젝트 운영 중 지속)

```
1. Evaluator가 evaluation-report.md를 생성
        ↓
2. 사용자가 보고서를 검토
        ↓
3. Evaluator의 판단 vs 사용자의 판단 비교
        ↓
4. 불일치 사례를 calibration-log.md에 기록
        ↓
5. 불일치 패턴에 따라 evaluator.md 또는 evaluation-criteria.md 업데이트
        ↓
6. 다음 프로젝트에서 개선 확인
        ↓
(반복)
```

---

## 단계별 실행 방법

### Step 1: 평가 보고서 검토

Evaluator가 생성한 `evaluation-report.md`를 직접 읽는다.
동시에 앱을 직접 열어보고 자신의 판단을 형성한다.

### Step 2: 불일치 식별

다음 유형의 불일치를 찾는다:

| 불일치 유형 | 예시 |
|------------|------|
| **관대 채점** | Evaluator가 8/10을 줬는데, 실제로는 명백한 버그가 있음 |
| **과잉 감점** | 사소한 이슈에 큰 감점을 적용함 |
| **놓친 문제** | 보고서에 언급되지 않은 명백한 이슈가 있음 |
| **표면적 테스트** | "검색이 동작한다"고 했지만 실제로는 기본 쿼리만 테스트함 |
| **자기 설득** | "이 이슈는 사소하므로 감점 없음"이라고 했지만, 데모에서 분명히 보임 |
| **독창성 과대평가** | 전형적 AI 패턴인데 "독창적"이라고 평가 |

### Step 3: calibration-log.md에 기록

`projects/{name}/calibration-log.md`에 불일치 사례를 기록한다:

```markdown
## 캘리브레이션 로그

### 프로젝트: {name}, 라운드: {N}

#### 불일치 1: 관대 채점
- **Evaluator 판단**: 디자인 품질 8/10 — "시각적 응집력 양호"
- **실제 관찰**: 메인 페이지와 서브 페이지의 색상 톤이 완전히 다름
- **교정**: "색상 톤이 페이지 간 일치하는지" 기준을 evaluation-criteria.md에 명시

#### 불일치 2: 놓친 문제
- **Evaluator 판단**: 기능성 검증에서 검색을 "PASS" 처리
- **실제 관찰**: 빈 검색어로 검색 시 전체 결과가 아닌 에러 표시
- **교정**: evaluator.md에 "각 기능별 빈 입력 테스트 필수" 지시 강화
```

### Step 4: 프롬프트/기준 업데이트

불일치 패턴에 따라 다음 파일을 수정한다:

| 불일치 패턴 | 수정 대상 | 수정 방법 |
|------------|----------|----------|
| 관대 채점 반복 | `prompts/evaluator.md` | 회의적 테스트 지시 강화, 구체적 감점 트리거 추가 |
| 특정 기준 미적용 | `templates/evaluation-criteria.md` | 해당 기준의 검증 방법을 더 구체적으로 명시 |
| 독창성 오판 | `templates/evaluation-criteria.md` | 감점 트리거 패턴 목록 확장 |
| 엣지 케이스 미탐사 | `prompts/evaluator.md` | 기능별 최소 테스트 시나리오 수 지정 |
| 스크린샷 부족 | `prompts/evaluator.md` | 필수 스크린샷 목록 명시 |

### Step 5: 개선 확인

다음 프로젝트(또는 같은 프로젝트의 다음 라운드)에서:
- 이전에 발견된 불일치 유형이 해소되었는지 확인
- 새로운 불일치가 발생했는지 확인
- 전반적 채점 정확도가 개선되었는지 평가

---

## 초기 캘리브레이션 (첫 3개 프로젝트)

첫 3개 프로젝트 동안은 **모든 평가 보고서를 직접 검토**하는 것을 권장한다.
이 기간에 Evaluator의 주요 편향이 드러나고, 프롬프트 튜닝의 80%가 완료된다.

### 집중 관찰 항목

1. **독창성 채점 패턴**: Claude가 자신이 생성한 전형적 패턴을 "독창적"이라고 하는지
2. **관대함 수준**: 전반적으로 점수가 높은지 낮은지 (70점 합격선 대비)
3. **테스트 깊이**: 기능당 몇 가지 시나리오를 테스트하는지
4. **수정 지시 품질**: Generator가 수정 지시만으로 문제를 해결할 수 있는지

## 캘리브레이션 성숙도 지표

| 수준 | 기준 | 상태 |
|------|------|------|
| Level 0 | 캘리브레이션 미실행 | 초기 상태 |
| Level 1 | 3개 프로젝트 검토 완료, 주요 편향 교정 | 기본 사용 가능 |
| Level 2 | 10개+ 프로젝트, 불일치율 20% 미만 | 안정적 |
| Level 3 | 사용자 검토 없이도 일관된 채점 | 자동화 신뢰 가능 |

---

## 주의사항

- 캘리브레이션은 **한 번에 끝나지 않는다**. 프로젝트 유형, 기술 스택, 디자인 요구가 달라지면 새로운 편향이 나타난다.
- **새 모델 출시 시** (예: Opus 새 버전) 캘리브레이션을 리셋하고 다시 시작한다. 모델 변경은 채점 패턴을 바꾼다.
- 캘리브레이션 로그는 지우지 않는다. 시간 경과에 따른 Evaluator 정확도 추세를 파악하는 데 유용하다.
